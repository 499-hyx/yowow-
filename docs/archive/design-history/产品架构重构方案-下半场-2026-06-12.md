## 1. 当前系统最大问题：直接给结论

**卡点不是热点、不是分析、也不是账号记忆。卡点有两个，按严重程度排：**

**第一卡点：系统在「生成可发草稿」处戛然而止，发布之后是真空。**
全链路终点是 `data/today/<acct>/latest.json` + 一键复制按钮。这条内容有没有发、什么时候发的、标题改没改、播放多少、几条私信、有没有人问价——系统一概不知。现有两套反馈组件（`FeedbackV1Box` 三项打分、`FeedbackBar` 六维打分）收的全是**老板对草稿的审美评价**，不是市场结果。审美反馈能修文风，修不了生意。这是「反馈闭环缺失」，但更准确地说是：**现实世界的数据没有任何进入系统的通道**。

**第二卡点：教育赛道在系统里没有「可成交的产品对象」。**
`config/tracks/education-yowow.json` 的 `product_value` 是一句能力主张（"孩子需要在陌生场景重新抓结构…"），`commercial_goal` 是建立信任/引流/品牌心智——没有 SKU、没有价格、没有交付形态、没有承诺边界。你在需求里提到的**状元机/拍学机，在整个仓库中零次出现**（仅 doctrine 中有"壮游机"作为产品锚，且只是分析落点不是销售对象）。话术系统的前提是"有东西可卖、知道它卖多少钱、知道买它的人在跟什么比"。**这块现在是空的，AI 推理填不了，必须人来补产品事实。**

**逐项排除（为什么其他不是主要矛盾）：**

- 热点不够？不是。06-12 两池共 24 条，剃须刀 17 条判定里 strong_pick 只取 2 条——供给远大于消化能力。
- 分析不够？不是。`analysis_doctrine`（step_zero_gate、超然观点公式、不碰清单）+ 五步桥梁链是这个系统最厚的部分，质量也确实在线（06-12 剃须刀的"医生科普→流程产品化"桥是自然的）。
- 账号记忆不够？半对。业务事实字段（卖什么/给谁/焦虑/证据）齐了，但缺的全是**成交侧**字段：评论区真实问题、成交阻力、转化目标、痛点的"原话版"。详见 §3。
- 平台表达没分层？**这个判断不准确**。`config/platforms/*.json` 五个平台各有 hook/title_logic/rewards/penalizes/conversion_path/expression_rules，且通过 `{platform_json}` 注入 generate 模板——分层已存在。真正缺的是：①一条素材只产出绑定平台的一个版本，没有"同素材跨平台多版本"；②platform 配置里的 `conversion_path`（如抖音"购物车挂载+评论区引导+私信"）在生成产物中没有对应输出字段——平台知道怎么转化，内容卡不知道。
- 销售转化链路缺失？是，且和第一、第二卡点是同一件事的三个面。

**对你方向的一个直接批评**：你说"缺市场反馈反向训练系统"——方向对，但当前体量（2 个账号、零条已记录的发布、零条市场数据）谈"反向训练"为时尚早。先要的是**纸面台账**（每天 5 分钟手填），数据多了才谈得上"训练"。先建自动化反馈系统是典型的过度工程。

## 2. 新系统全链路设计

升级目标链路：`账号定位 → 用户痛点 → 热点选择 → 产品桥接 → 平台表达 → 销售话术 → 发布反馈 → 反向迭代`

设计原则（Agent 架构视角）：**不新建引擎，每一步都挂在现有文件接口上**；方法论进 track 文件、业务事实进 account 文件、判断进 prompt 可编辑区、产物进 data/——这是本仓库已验证的分层，下半场沿用，不发明新形态。

| 步骤 | 输入 | 输出 | 谁维护 | 存哪里 | 前端展示 | AI 介入 | 人介入 |
|---|---|---|---|---|---|---|---|
| ① 账号定位 | onboarding 7 问 + 新增成交侧 6 问（§3） | AccountMemory（扩展版） | 老板/运营答，运营更新 | `data/accounts/<id>.json`（现有文件，加可选字段） | 账号记忆页/账号主页（现有，加区块） | 起草建议值 | 确认、改写、定稿 |
| ② 用户痛点 | 账号记忆 anxiety_anchors + 评论区/私信原话 + 话术库 FAQ | `pain_points[]`：每条 = 痛点 + 家长原话 + 出现场景 + 强度 | 运营每周摘录，博士审教育线 | account memory 新字段（账号级）+ 话术库（赛道级共享） | 账号主页"用户痛点"卡 | 从原话聚类提炼草稿 | 摘原话的是人；强度判断是人 |
| ③ 热点选择 | 两池热点 + 赛道 doctrine（现状不变） | match 回贴（tier + 理由） | 博士改 match 可编辑区 | `data/runs/<D>/.../match-*.json`（现状） | 首页矩阵/热点池（现状） | 全部判定 | 博士改判断口径；maybe 拍板 |
| ④ 产品桥接 | strong_pick/maybe 热点 + 账号记忆 + 痛点 | bridge_paths 五步链（现状不变） | 博士定方法论 | AdaptationOutput（现状） | 推荐卡"几种讲法"（现状） | 全部生成 | 抽审 |
| ⑤ 平台表达 | bridge + platform_json（现状） | content（title/body）；**新增**：跨平台变体（仅对必发条，按需生成） | 老板改口吻区 | AdaptationOutput.content；变体存 runs | 推荐卡草稿区 | 全部生成 | 编辑、定稿 |
| ⑥ 销售话术（临门一脚） | chosen bridge_path + 痛点 + 话术库 top-k 相关条目 + 平台 conversion_path | `closer` 对象（§4 定义） | 话术库：运营+销售沉淀，博士审；生成：AI | Phase 2 起进 AdaptationOutput 可选字段；话术库在 `config/playbook/<track>/` | 推荐卡新区块"怎么接到买" | 按话术库素材组装，**不许编造案例** | 话术库条目全部来自真人；成交承诺红线人定 |
| ⑦ 发布反馈 | 实际发布动作 + 平台后台数据 + 私信/留资/成交记录 | 发布台账条目（§5 字段） | **运营手填**，每天 5 分钟 | `data/publish-log/<acct>/<月>.json`（新目录，网站可写或手填） | 账号主页"已发布"区 + 内容卡状态标 | 无（先不要自动抓数） | 全部 |
| ⑧ 反向迭代 | 发布台账 + 反馈打分 + 话术库胜败案例 | 周复盘：「强化清单 / 避免清单 / 话术库增补建议」 | agent 每周渲染复盘模板，老板+博士拍板 | `data/runs/<D>/_retro/`；拍板结论写回 account memory 与话术库 | 账号主页"上周结论"卡 | 起草复盘 | **写回必须人批**，绝不自动调权 |

红线延续：⑥ 的成品同样过禁词硬门和"牵强即 skip"——**桥接不自然的条目连临门一脚都不配有**，不能为了转化把 skip 捞回来。

## 3. 账号记忆板怎么改

现有 `AccountMemory` 保留不动（business/audience/product_value/anxiety_anchors/proof_assets/commercial_goal/content_style/extra_*/banned_topics）。新增以下可选字段，全部 additive，不破坏 `loadDataAccounts()` 现有读取：

| 新字段 | 形态 | 为什么需要（不是凑数） |
|---|---|---|
| `account_role` | 枚举：科普/销售/IP/老师/品牌 | 现在 positioning（boss/expert）只管口吻，不管**这个号在转化链里的分工**。矩阵打法里科普号和销售号发同一热点的写法完全不同：科普号讲原理建信任，销售号给"今天就买"的理由。没有这个字段，⑥ 临门一脚不知道该收口到"关注我"还是"拍下链接"。 |
| `pain_points[]` | `{pain, customer_quote, scene, intensity}` | `anxiety_anchors` 是老板视角的概括（"怕状态差"）；卖货需要的是**家长/用户原话**（"孩子三年级了天天问 AI 作业答案，我该不该断网？"）。原话才能进标题、进钩子、进评论区回复。概括只能写文案，原话才能写出"这说的就是我"。 |
| `core_selling_points[]` | `{point, vs_alternative, proof_ref}` | `product_value` 是一句话主张；卖点需要"跟什么比"。家长买状元机不是跟空气比，是跟学而思网课比、跟自己陪读比、跟 iPad 比。没有 vs_alternative，话术只能自夸，无法解除比较犹豫。 |
| `faq[]` | `{question, good_answer, source}` | 评论区高频问题是免费的市场调研，现在全靠运营脑子记。沉淀下来后：生成时预埋答案到正文、评论区直接复制回复。source 标"评论区/私信/直播"，防 AI 编造。 |
| `conversion_goal` | 枚举+自由文本：挂车成交/私信留资/进群/关注 | 没有它，⑥ 的收口动作只能瞎猜。剃须刀号是挂车直购，教育号是私信→老师承接——两者的"临门一脚"完全不同。 |
| `conversion_friction[]` | `{friction, counter}` | 成交阻力（"太贵""孩子会不会玩物丧志""跟免费 App 有啥区别"）+ 已验证的化解话术。这是销售转化专家视角下**整个下半场最值钱的字段**，且只能由真实销售/老师喂进来。 |
| `platform_tone_note` | 自由文本 | platform 配置是平台通用规则；这个字段记**这个号在这个平台被验证过的个性**（"我们号的粉丝吃工厂实拍，不吃剧情"）。来自 ⑧ 复盘写回，不是拍脑袋填。 |
| `reinforce_list[]` / `avoid_list[]` | 字符串数组，带日期 | ⑧ 周复盘的落点。"下次强化/下次避免"必须有归宿才不会变成口头禅，归宿就是这两个数组，generate 渲染时注入。 |

**维护权责**：`pain_points/faq/conversion_friction` 的素材来源是人（运营摘评论区、销售记私信），AI 只做聚类整理草稿；`account_role/conversion_goal` 老板定；`reinforce/avoid` 复盘后人批写入。

**一个直话**：教育账号（小朱号）填这套字段前，必须先回答"卖什么、多少钱、怎么交付"。当前 track 文件里 commercial_goal 是建立信任——如果近期就是不卖货只蓄水，那 `conversion_goal: "私信留资"` 也是合法答案，但要明确写下来，不能空着让 AI 猜。

## 4. 热点分析怎么改：「临门一脚」模块（closer）

定位：**不改 match、不改五步桥梁链**。closer 是 generate 之后的追加产物，只对 strong_pick / maybe 生成（skip 永远没有 closer）。

### closer 对象设计（Phase 2 进 AdaptationOutput 可选字段）

```text
closer: {
  product_entry:     这个热点场景里产品自然出场的那个瞬间是什么
                     （例：讲完"刀片不换会发炎"，出场点是"换刀周期"而不是品牌名）
  buy_trigger:       用户此刻为什么会产生购买动机——场景里的哪个具体时刻
                     （"明早刮脸时会想起这条视频"才是动机，"觉得有道理"不是）
  sell_lever:        四选一+理由：卖功能 / 卖情绪 / 卖安全感 / 卖未来收益
                     （剃须刀多数卖"确定性"=安全感；教育卖未来收益最容易翻车成贩卖焦虑，
                      doctrine 明令禁止，教育线默认卖"家长的掌控感"=安全感）
  platform_lines: {
    xiaohongshu:     收口写法——经验分享体，转化藏在"我怎么选的"清单里，评论区接问题
    douyin:          收口写法——口播最后 5 秒行动指令 + 评论区置顶 + 挂车话术
    shipinhao:       收口写法——熟人语境，"转给老公/孩子他爸看"式社交货币 + 私信引导
  }
  hard_sell_warnings: [ 这条热点上哪些说法会显得硬蹭/违禁，逐条列 ]
                     （例：防晒虚标热点上直接说"买我们的就不虚标"= 硬蹭+违广告法）
}
```

### 为什么这样设计（逐问回答你的要求）

- **如何自然引出产品**：`product_entry` 强制回答"出场瞬间"，且必须长在 chosen bridge_path 的第 4 步（product_value_support）上——桥不自然，closer 不许存在。这是防硬蹭的结构性约束，不靠自觉。
- **购买动机**：`buy_trigger` 强制写"场景里的时刻"而不是"卖点复述"。判据：用户在自己生活里**再次遇到这个场景时**会不会想起这条内容。
- **卖什么**：`sell_lever` 强制四选一，逼 AI 表态而不是四个都沾。教育线加硬约束：**禁选"卖未来收益"中的恐吓式表达**（"不学就落后"），这与 doctrine 的"不贩卖焦虑"对齐——转化设计必须服从教义，冲突时教义赢。
- **三平台说法**：直接消费 `config/platforms/*.json` 已有的 `conversion_path` 和 `expression_rules`（抖音=购物车+评论区+私信；小红书/视频号各异）——配置已存在，closer 只是第一次真正用它产出。
- **哪些说法硬蹭**：`hard_sell_warnings` 是负面清单，渲染进卡片让运营发布前自查；同时是 gate 的检查素材。

### 落地路径

Phase 1（不动 schema/prompts）：closer 不进产物，由 agent 在每日汇总时对必发条**手工渲染追加段落**，存 `data/runs/<D>/<acct>/closer-<hotspot_id>.md`，前端不展示，先验证内容质量。
Phase 2：新增 `prompts/closer.md`（新文件，additive，可编辑区博士/老板定）+ AdaptationOutput 加可选 `closer` 字段。**注意：这是 schema 变更，按仓库铁律须走工单 + 人拍板，不能顺手加。**

## 5. 反馈系统怎么建：发布台账 + 三条回流路由

### 发布台账（核心新增物，Phase 1 就建）

位置：`data/publish-log/<account_id>/<YYYY-MM>.json`（新目录，不碰 data/today 铁律）。每条：

```text
{
  publish_id, account_id, platform, published_at,
  hotspot_id,            ← 锚回热点池（素材证据链延伸到发布侧）
  final_title,           ← 实际发布标题（可能被运营改过，diff 本身就是信号）
  content_direction,     ← 科普/测评/观点/剧情
  selling_point_used,    ← 用了哪个卖点（对 core_selling_points 的引用）
  closer_used,           ← 用没用临门一脚、用的哪个 sell_lever
  metrics: { views, ctr, completion_rate, interactions, dm_count, leads, deals },
                         ← 全部可空。有什么填什么，没有就空着，不许编
  human_review,          ← 一句话人工评价
  reinforce_next,        ← 下次强化什么
  avoid_next             ← 下次避免什么
}
```

**录入方式直说**：前期就是运营每天花 5 分钟手填（前端一个表单写文件，或直接给 agent 报数）。**不要先做平台 API 自动抓取**——抖音/小红书开放接口拿不全这些数，爬虫不稳定还有封号风险，且当前发布量（每天 2-3 条）完全不值得自动化。

### 三条回流路由（全部人批，无自动调权）

1. **→ 账号记忆**：周复盘把 `reinforce_next/avoid_next` 聚合成草稿，老板确认后写入 memory 的 `reinforce_list/avoid_list`；generate 渲染时注入，下周生成直接带着上周教训。`human_review` 中关于号性的发现（"粉丝吃实拍"）落 `platform_tone_note`。
2. **→ 热点选择**：复盘发现的题材规律（"测评类完播率持续垫底""争议观点类互动 3 倍"）由**博士手动**吸收进 match 模板可编辑区或 track 的 `analysis_doctrine`——走"改模板可编辑区"的现有机制，不新建通道。频率最多每月一次，防止小样本噪声污染判断标准。
3. **→ 话术库**：`deals > 0` 的条目沉淀为 win_case（带当时的标题、卖点、closer 写法）；明确翻车的沉淀为 fail_case。话术库条目的 `confidence` 字段据此升降。

**为什么坚持人批**：当前样本量下任何"自动反向训练"都是在拿噪声当信号。两个账号每周不到 20 条发布，统计上不支持自动权重调整；人批的成本极低（每周一次 15 分钟），且符合本仓库"判断进可编辑区、人拍板"的既有治理结构。

## 6. 销售话术库怎么建

位置：`config/playbook/<track_id>/phrases.json`（赛道级共享，账号可通过 memory 引用补充）。放 config 而非 data 的理由：它和 example_bridges 一样是"人沉淀的判断素材"，进提示词装配，不是每日流水。

### 条目结构（七类一个 schema）

```text
{
  id, type,              ← sales_script / teacher_explain / faq / objection /
                            win_case / fail_case / style_sample
  scene,                 ← 什么场景下用（评论区/私信/直播间/正文预埋）
  customer_words,        ← 客户/家长原话（必填，这是整个库的灵魂）
  our_words,             ← 我们的应对原话（销售/老师真实说过的版本）
  outcome,               ← 当时结果：成交/留资/流失/未知
  source,                ← 谁、哪天、哪个渠道（防编造的溯源字段）
  style_tag,             ← 直给型/陪伴型/专家型——不同销售风格并存，不强行统一
  confidence             ← 高/中/低，由 ⑧ 回流升降
}
```

**为什么七类同库**：老师解释话术和销售话术在教育产品里经常是同一句话的两种温度；分库会逼运营做无谓的分类决策。用 `type` + `style_tag` 标注即可。

### AI 怎么调用（关键：机械检索，不做向量库）

`make-prompt` 装配 generate/closer 提示词时，按当前热点命中的 `anxiety_anchor` + 关键词匹配，取 top 3-5 条相关条目注入提示词（"以下是真实话术参考，可改写语气，**不得虚构其中没有的案例和数字**"）。检索就用关键词包含匹配——库规模在几百条以内，向量检索是过度工程。

**硬规则**：AI 只许改写 `our_words` 的语气与平台形态，**不许发明 customer_words、不许编造 outcome**。话术库为空时 closer 照常生成但标注"无真实话术支撑，发布前人工核"——诚实降级，不假装有积累。

### 冷启动（直说这是最大风险）

库的第一批条目只能来自：小朱的评论区私信存量、博士给家长讲解的真实记录、剃须刀客服聊天记录。**如果团队拿不出 30 条真实记录，这个模块就不该启动**——空库 + AI 推理 = 把"AI 编的话术"洗成"沉淀的经验"，比没有更危险。

### 第三阶段 · 反馈回流（持续运营，每周一次）

1. 新建 `prompts/weekly-retro.md`（additive）：输入 = 本周 publish-log + feedback 文件，输出 = 强化/避免/话术库增补三张草稿清单。

## 8. 一句产品判断

**这个系统应该是「内容成交辅助系统」，热点分析只是它的引擎——但"成交"在你的两条赛道里含义不同，混为一谈会翻车**：剃须刀线的成交是挂车直购，话术可以收口到"今天就换"；教育线（博士 IP + 高客单 + 信任型决策）的成交是"内容建信任 → 私信/评论承接 → 老师转化"，内容端硬塞购买理由会同时毁掉博士教义（不贩卖焦虑、超然观点）和转化率本身。所以方向修正一句话：**下半场不是给内容加销售话术，而是给"发布之后"补上台账、承接和复盘——话术长在承接环节，不长在正文里。**

另：你提的状元机/拍学机在仓库中无任何产品资料（标注：不确定这两个产品的形态/价格/交付），第一阶段第一件事就是把它们的产品事实补进 track/account 文件，否则 §4 §6 对教育线都是空转。
