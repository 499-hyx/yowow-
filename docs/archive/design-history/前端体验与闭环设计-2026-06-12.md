## 0. 「前端卡卡的」病根诊断（基于文件证据，不是猜）

| 病根 | 证据 | 后果 |
|---|---|---|
| ① 全站零缓存 | 所有页面 `force-dynamic`；`lib/turso.ts:26` fetch 写死 `cache: "no-store"`；全仓库无 `revalidate`/`React.cache` | 每次点击都全量重读，哪怕数据一天只变一次 |
| ② Turso 模式下串行 HTTP 风暴 | `getDoc`/`listDocKeys` 每次调用 = 一次独立 Turso HTTP 往返。首页 `buildDashboardSnapshot` 串行发 10+ 次；账号页 `loadAccountHistory`（dashboard-data.ts:280-292）**for 循环里逐个 await**，30 天历史 = 30 次串行往返 ≈ 3-9 秒 | 线上账号页最卡的就是它 |
| ③ 无 loading.tsx | `app/` 下只有 today/memory/onboarding 有 layout，**没有任何 loading.tsx** | 点链接后白屏等服务端，体感"死了" |
| ④ 重复查询 | `resolveDateContext` 和 `loadHotspots` 各自调 `listHotspotDates`，首页一次渲染同一列表查 2-3 遍 | 白白翻倍 |

### P0 修复（下一轮第一件事，改 4 个文件）

1. **`lib/data-source.ts`**：`getDoc`/`listDocKeys`/`listDocs` 用 React `cache()` 包一层（请求内去重，修病根④，零行为变化）。
2. **`lib/dashboard-data.ts`**：
   - `loadAccountHistory` 改为 `listDocs("today", "<acct>/")` 一次取回全部（Turso 模式 1 次查询代替 N 次），本地模式同接口；历史默认只取最近 14 天。
   - `buildDashboardSnapshot` 里 `loadHotspots / loadDataAccounts` 用 `Promise.all` 并行。
3. **读页缓存策略**：四个读页（`/`、`/hotspots`、`/account/[id]`、`/card/...`）把 `force-dynamic` 换成 `export const revalidate = 60`。数据只在跑批/sync 时变，60 秒陈旧完全可接受；页面顶部已有生成时间，不骗人。写操作（feedback/发布登记/记忆保存）完成后前端调 `router.refresh()` 即时刷新。
4. **新增 `app/loading.tsx` + `app/account/[account_id]/loading.tsx` + `app/hotspots/loading.tsx`**：骨架屏（仿 TodayBoard 的 `animate-pulse` 块，每个文件 ~15 行）。点击即有反馈，体感秒开。

预期效果：本地几乎无感；线上账号页从秒级降到 <500ms，且二次访问命中缓存。**先做这个再谈任何新功能，否则新页面一样卡。**

## 1. 信息架构：每个账号一个专栏页

现状问题：账号相关功能撒在 4 个地方（`/account/[id]` 看推荐、`/memory?account=` 只读记忆、`/today?account=` 旧今日页、`/accounts` 列表），改记忆要手编 JSON。整合为：

```
/account/[account_id]            ← 账号专栏（唯一入口，tab 切换不换页）
  ?tab=today      今日推荐（默认）
  ?tab=memory     账号记忆（可编辑）
  ?tab=spark      灵感收件箱
  ?tab=history    历史与反馈
```

- Tab 用 searchParams 驱动（服务端组件友好，可直链可收藏）。
- 页头固定：账号名 + 赛道/平台/人设 + 记忆完整度（复用 `lib/memory-meta.ts` 的 `memoryCompleteness`）+ 跑批状态点（今天有/无 latest）。
- `/memory` 与 `/today` 保留路由但 302 跳转到对应 tab（不删代码，防外链断）。
- 首页账号行点击 → 直达专栏 today tab。

### tab=today（现有内容重排，不重写）

保留这轮做好的速览列表 + 详细卡 + 原素材证据块。两个体验改动：
1. 详细卡默认 `<details>` 折叠（上轮审计 P2：速览和全卡重复导致页面过长），必发条默认展开。
2. 速览行点击行为改为展开对应卡（锚点跳转保留为 fallback）。

### tab=memory（本轮核心，见 §2）
### tab=spark（见 §3）
### tab=history（现有历史表 + 已有反馈记录合并展示，Phase 2 再加发布台账）

## 2. 账号记忆：从「手编 JSON」到「页面内编辑」

### 交互设计

- 记忆按现有 `memorySections` 分组渲染为卡片；每张卡右上「编辑」进入行内编辑态（文本字段 textarea、数组字段 chips 输入：回车添加、点 × 删除）。
- 保存即整卡提交；成功 toast「已保存，下次跑批生效」+ 头部完整度即时更新；失败给人话原因。
- 字段旁灰字提示语保留 onboarding 七问的问法（"你的客户最焦虑什么？"），老板不需要理解字段名。
- **保存防线（必须有）**：
  1. 提交前后端都跑 `scanInternal`（adaptation-types.ts 已有）——记忆里写进"远迁移/OOD"直接拒绝并标红，防止老板亲手污染提示词装配。
  2. 服务端写文件前备份 `data/accounts/.bak/<id>.<timestamp>.json`（保留最近 5 份）。
  3. 写入时盖 `memory_updated_at`（字段已存在于 StoredAccount，一直没人写）。

### API 契约（新增，模式照抄 feedback v1）

```
PATCH /api/accounts/[account_id]/memory
body:  { memory: Partial<AccountMemory> }     ← 只传改动的 section
逻辑:  读现有 account json → 浅合并 memory → scanInternal 全字段 →
       备份 → json.dump(ensure_ascii 等价: JSON.stringify 中文原样) → 写回
返回:  { ok, memory_updated_at } 或 { ok:false, error, hits:[命中的禁词] }
```

**双源写入的诚实处理**（这是关键决策点）：本地文件是事实源（DEPLOY 架构已定）。
- 本地/自部署模式：直接写 `data/accounts/<id>.json`，立即生效。
- Vercel 生产模式（tursoEnabled）：**不直接写 docs 表**（会倒置事实源），照抄 feedback_inbox 模式写 `memory_edit_inbox` 表，提示文案改为「已提交，下次同步生效」；`pull-feedback.py` 的同构脚本拉回本地合并。Phase 1 可以只做本地模式 + 生产模式显示「线上暂不支持编辑，请在本地改」——诚实降级好过悄悄分叉事实源。

## 3. 灵感收件箱（owner spark）：灵光一现怎么进系统

你的例子（"高考后数学能力的退化是怎样的神经科学过程→6 月高考期间小红书爆文选题"）本质是：**老板自命题，希望它和当日热点一起进入筛选/生成流水线**。设计原则：spark 是赛道池的一条进货渠道（和外部 GPT 报告同级），**不绕过中立化和 ingest 铁律，也绝不入公共池**。

### 数据流（和 AGENT-PLAYBOOK 入口①衔接，记为「入口①c」）

```
网页提交 → data/spark-inbox/<account_id>/<spark_id>.json   (status: pending)
                ↓ 第二天 agent 跑「抓今天的热点」时顺带处理：
   读 pending spark → 当作定向检索线索跑搜索（找外部支撑素材/相关报道）
     ├─ 找到外部素材 → 正常走 neutralize 富化 → 入该赛道池
     │    source_direction: "owner-spark:<spark_id>"，有 source_url
     ├─ 纯自命题（如神经科学设问，无新闻事件）→ 仍经 neutralize 整形入赛道池，
     │    source_skill: "owner-spark"，source_url 可空——这是「无链接不入池」原则的
     │    一个显式例外，前端对这类条目显示「自命题」徽章而不是「暂无原文链接」
     └─ 既查无实据又撑不起剖析 → spark 标 rejected + 人话原因（铁律：缺料不捏造）
   处理完盖章 status: ingested/rejected + 关联 hotspot_id
```

spark 文件格式：

```json
{ "spark_id": "spark-20260612-001", "account_id": "...", "track_id": "...",
  "text": "老板原话，整段保留", "created_at": "...", "status": "pending",
  "resolved_at": null, "hotspot_id": null, "reject_reason": null }
```

### 前端交互

- 专栏 spark tab：顶部一个大 textarea（占位文案："想到什么写什么，明天跑热点时会自动带上"）+ 提交按钮；下方列表按状态分组：待处理（黄）/ 已入池（绿，链到 `/hotspots/<id>`）/ 未采用（灰，显示原因）。
- 首页右上加「+ 灵感」快捷按钮 → 弹出账号选择 + textarea（或直接跳专栏 tab）。
- **诚实边界**：网页只负责收，处理发生在第二天 agent 跑批时（或你对 agent 说"处理灵感收件箱"）。tab 里写清楚这句话，不许暗示"提交即生效"。

### 改动文件

`app/api/spark/route.ts`（新，POST 写文件 + GET 列表）、专栏页 spark tab、`AGENT-PLAYBOOK.md` 加入口①c 一节（约 10 行，这是文档不是 prompts 正文，但改前向你确认）。

## 4. 首页闭环：今日流水线状态条

你每天的固定动作是：跑公共池 → 跑各赛道池 → 各账号跑批 →（偶尔）处理灵感。现状是首页只有结果数字，**看不出"今天还差哪步"**。新增状态条（首页 DateContextBar 下方一行）：

```
今日流水线   公共池 ✓ 12条·09:30   教育池 ✓ 15条   剃须刀池 ✗ 未抓
            跑批 1/2（小朱号未跑）   灵感待处理 2
```

- 检测逻辑全是现有文件存在性判断（`data/hotspots/<今天>.json`、`tracks/<id>/<今天>.json`、`data/today/<acct>/<今天>.json`、spark pending 计数），零新依赖。
- 每个 ✗ 项可点开，给一句**可复制的 agent 话术**（"抓今天的热点""剃须刀号今天发什么""处理灵感收件箱"）——网站不能替你跑 LLM 批次，这是架构事实；闭环的诚实形态是"状态可见 + 命令一键复制"，不是假装有个按钮能跑。
- 实现：`lib/pipeline-status.ts`（新，~60 行纯函数）+ 首页一个 section。
