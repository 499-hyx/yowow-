# 任务：为一条新赛道起草「桥梁母题」

今天是 {date}。下面是这条新赛道老板的业务种子，请据此起草这条赛道的桥梁母题。

## 业务种子（来自老板的 onboarding 回答）

- 卖什么 / 给谁：{business_seed}
- 产品最大的好：{product_value}
- 能拿出来取信的证据：{proof_assets}
- 客户最焦虑 / 最在意的：{anxiety_anchors}

## 起草方法论（博士在此编辑——这是系统真正的「脑子」）

> 下面这段是**指导 LLM 怎么起草**的方法论。博士可以自由改写、加原则、加反例。
> 引擎不碰这段，它只是文本。改这里 = 改整条赛道的连接哲学。

起草这条赛道桥梁母题时，请遵循以下原则：

1. **先看热点这一端、再看赛道这一端，桥要自然长出来，不是硬凑。**
   一条好的桥梁母题，是「这类热点天然会戳中客户的某种真实焦虑，而这种焦虑正好是本赛道
   产品能回应的」。如果要绕很远才接得上，那不是母题，是牵强。

2. **区分「内部理解锚」和「对外表达词」。**
   - 内部理解锚（`internal_lens`）：是博士判断「这条热点能不能承载本赛道主张」的专业视角，
     可以用专业术语（例如教育赛道用的 far transfer / 远迁移）。它**只在后台用来判断自然度，
     永远不出现在给用户看的内容里**。
   - 对外表达词（`external_vocab`）：是把上面那个专业视角**翻译成受众听得懂的人话**。
     例如把 far transfer 翻成「会面对陌生问题的能力」「会提问比会答更重要」。

3. **禁词（`forbidden_terms`）= 把内部锚里那些黑话列出来，明令对外不准出现。**
   至少列出内部锚里的核心术语，确保它们不会泄漏到成品。

4. **example_bridges 给 2~4 条**，每条写清：什么样的热点（hotspot_hint）→ 戳中什么真实问题
   （real_problem）→ 和本赛道什么关系（track_relation）。这是给后续生成环节的 few-shot 种子。

## 输出格式（严格输出下面这个 JSON，不要多余文字）

```json
{
  "track_name": "<给这条赛道起一个人话名字>",
  "buyer_job": "<一句话：老板最想让受众相信什么，从而认可他的内容/产品>",
  "product_value": "<把产品最大的好提炼成一句>",
  "anxiety_anchors": ["<客户焦虑1>", "<客户焦虑2>", "..."],
  "bridge": {
    "internal_lens": "<后台判断视角，1 句，专业术语可用，永不外显>",
    "external_vocab": ["<对外人话词1>", "<对外人话词2>", "<对外人话词3>", "..."],
    "forbidden_terms": ["<对外禁词1>", "..."]
  },
  "example_bridges": [
    { "hotspot_hint": "<什么样的热点>", "real_problem": "<戳中的真实问题>", "track_relation": "<和本赛道的关系>" }
  ]
}
```

## 护栏（必须满足，否则起草不合格）

- `external_vocab` ≥ 3 条、`forbidden_terms` ≥ 1 条。
- `internal_lens` 里的术语**不得**同时出现在 `external_vocab` 里（内部锚不能当对外词用）。
- 不要发明客户没说过的业务事实；起草不确定的地方，宁可写得保守、留给博士定稿。
