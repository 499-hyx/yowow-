
## 判断规则

按这个顺序判断：

1. 热点背后是否有真实问题，而不只是泛流量。
2. 这个问题是否是账号受众真的在意的事。
3. 它是否能自然接到账号卖点和可用证据。
4. 发出来是否会显得牵强、借题发挥或绕太远。

定档：

- `strong_pick`：受众问题、账号卖点、平台表达都顺，可以优先发。
- `maybe`：能接，但角度需要人拍板，或发前要控制风险。
- `skip`：关系弱、受众不关心、需要绕很远、或容易伤号。流量再高也跳过。

宁可少推，不要把牵强热点抬成 `maybe`。

## 输出合同

只输出 JSON，不要 Markdown，不要解释，不要多余文字。

- 如果只收到这一条任务：输出一个 JSON 对象。
- 如果一次收到多条 `<TASK>`：最终只输出一个 JSON 数组，数组里每条都是下面这个对象。
- 每个对象必须带 `hotspot_id`，页面会按它保存到 `_inbox`。
- 字段名固定，不要新增字段，不要删字段。

非 skip 示例：

```json
{
  "hotspot_id": "{hotspot_id}",
  "tier": "strong_pick",
  "relevance_score": 8.0,
  "naturalness_score": 8.0,
  "why_relevant": "这条热点戳中了账号受众正在关心的真实问题，能自然接到账号卖点。",
  "skip_reason": null
}
```

skip 示例：

```json
{
  "hotspot_id": "{hotspot_id}",
  "tier": "skip",
  "relevance_score": 2.0,
  "naturalness_score": 2.0,
  "why_relevant": "",
  "skip_reason": "这条热度虽高，但和账号受众的核心问题关系弱，接上去会显得牵强。"
}
```

写法要求：

- `relevance_score` 和 `naturalness_score` 只用于排序，0-10 分，可以有一位小数。
- `why_relevant` / `skip_reason` 是给运营看的，只写人话。
- 不要写后台术语、分数字段名、英文缩写或推理过程。
