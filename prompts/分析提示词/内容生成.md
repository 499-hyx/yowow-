## 生成方法

先判断，后写稿：

1. 先过博士分析层的第 0 步。命中不适合、细节不足、关系太弱，直接 `skip`。
2. 能接时，给 3 条不同的 `bridge_paths`。每条必须走完整 5 步：
   `phenomenon` → `real_problem` → `track_relation` → `product_value_support` → `platform_expression`。
3. 选最自然的一条，把它的 `path_id` 写进 `chosen_path_id`。
4. `content.body_or_script` 按赛道博士分析层展开，但对外只讲人话，不暴露后台术语。
5. 有争议或风险时，在 `risk_note` 给运营提醒；不要写进正文。

<!-- <<< 可编辑区 结束 ============================================================ -->

## 输出合同

只输出 JSON，不要 Markdown，不要解释，不要多余文字。

- 如果只收到这一条任务：输出一个 JSON 对象。
- 如果一次收到多条 `<TASK>`：最终只输出一个 JSON 数组，数组里每条都是下面这个对象。
- 每个对象必须带 `hotspot_id`，页面会按它保存到 `_inbox`。
- 字段名固定，不要新增字段，不要删字段。
- `recommendation` 只能是 `strong_pick`、`maybe`、`skip`。
- 非 skip 必须有 3 条 `bridge_paths`；skip 必须 `bridge_paths: []` 且 `content: null`。

skip 输出：

```json
{
  "hotspot_id": "{hotspot_id}",
  "relevance_score": 2.0,
  "naturalness_score": 2.0,
  "recommendation": "skip",
  "forced_flag": true,
  "skip_reason": "这条热度虽高，但和账号受众的核心问题关系弱，接上去会显得牵强。",
  "bridge_paths": [],
  "chosen_path_id": null,
  "content": null,
  "external_terms_check": true,
  "risk_note": null
}
```

硬性护栏：

- `content.topic`、`content.title`、`content.body_or_script` 里不能出现禁词。
- `bridge_paths` 和 `skip_reason` 里也不要出现后台术语。
- 不编造证据，不夸大效果，不写无法核验的承诺。
- 如果为了接上热点需要绕很远，直接 skip。
