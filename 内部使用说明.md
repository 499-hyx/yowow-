## 系统是干什么的

每天给各账号回答一个问题：**今天哪条热点能发、为什么能发、哪些别蹭。**

流程：抓热点 → 逐条判断与账号的匹配度 → 生成可用内容起草 → 人话汇总。全程走固定模板，判断逻辑可审计、可复查。

## 日常话术 → 动作对照

直接对 Claude（或 Codex）说这些话就行：

| 你说 | 系统做什么 |
|---|---|
| **「抓今天的热点」** | 跑大盘扫描 + 两个赛道定向搜索，写入热点池 |
| **「剃须刀号今天发什么」** | 给剃须刀账号跑筛选+内容生成，输出发什么/别蹭 |
| **「小猪号今天发什么」** | 给教育小红书号跑同样流程 |

| **「今天一条龙」** | 先抓热点，再给所有账号跑推荐，一次出完 |
| **「新增一个账号：……」** | 进入新建流程（先判断赛道存不存在） |
| **「给剃须刀赛道重写搜索方向」** | 重起草该赛道的搜索母题，改完需博士定稿 |

## 每日标准输出（人话汇总格式）

跑完每个账号，系统输出固定三段：

```
✅ 今天发什么：strong_pick 条目 + 一句话钩子
🤔 为什么能发：每条的桥梁逻辑一句话（不出现专业术语）
🚫 哪些别蹭：skip 条目 + 一句话原因
```

如有 `maybe` 条目，单列"可发但需老板拍板"。

## 角色分工

### 博士
- 审定新赛道的桥梁方法论（把 `status: draft` 改 `approved` 才能跑批）
- 审核 `maybe` 条目要不要发
- 修改各赛道的搜索方向和判断口径（改 `config/tracks/<id>.json` 可编辑区）

### 小朱 / 运营
- 每天看汇总输出，直接用 `strong_pick` 的内容起草
- 对 `maybe` 条目给是否发的意见
- 反馈哪条发了效果好/差（系统会记录）

### 小钟 / 技术
- 日常不需要操作，系统自动跑
- 新增账号时配置 `data/accounts/<id>.json`
- 遇到脚本报错看 `docs/RUNBOOK.md`

## 关键规则（运营需知）

1. **牵强就不发**——系统宁可全 skip 也不硬凑，skip_reason 会说明原因。
2. **内容起草里不会出现**"远迁移""far transfer""OOD"等内部词，看到了报给小钟。
3. **不要直接改 `data/today/` 里的文件**——只有系统脚本有资格写，手改会被下次覆盖。
4. **反馈给系统**：告诉 Claude「这条发了效果不好」或「这条老板不想发」，系统会记录并影响下次判断。

## 常见问题

**Q：能不能直接让 Claude 帮我写好一篇小红书？**  
A：可以。跑完推荐后，把 `strong_pick` 的内容起草（`bridge_paths` 里的 `platform_expression`）发给 Claude，说「帮我扩写成完整小红书图文」。

**Q：strong_pick 的钩子太专业，运营看不懂怎么办？**  
A：直接说「把这条解释得更通俗一点」，Claude 会重新用人话说。

**Q：想加一个新账号，比如宠物食品号？**  
A：直接说「新增一个账号：宠物食品，卖……」，系统会引导完成配置。赛道 petfood-sourcing 已起草，需博士定稿后才能跑批。

**Q：想看历史某天的数据？**  
A：历史归档在 `data/runs/<日期>/` 目录下，按账号分文件夹存放。
